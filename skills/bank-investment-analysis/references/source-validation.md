# 一手资料与验证规范

## 1. 资料包

财报点评的最小资料包：

1. 指定报告期的完整官方 PDF；
2. 上一期、上年同期、最近年报；
3. 业绩快报和业绩会实录（如有）；
4. 与结论直接相关的监管、央行、统计局原文；
5. 用于交叉验证的交易所结构化数据、银行 IR 演示或另一份官方披露。

不要用新闻摘要替代完整财报。若只有快报，只能做“快报点评”，并列出无法判断的附注项目。

### 目标报告锁定与停机规则

财报点评开始前先搜索银行官网投资者关系页面，再依次核对交易所/巨潮或港交所。下载并实际打开目标期完整PDF，建立候选清单后运行：

```bash
python scripts/resolve_bank_report.py --manifest <候选清单.json> --bank <银行全称> --code <证券代码> --period <2026Q1> --cutoff <YYYY-MM-DD> --require-local-pdf --output <锁定记录.json>
```

候选清单每项至少含 `bank_name`、`stock_code`、`report_period`、`title`、`publication_date`、`source_class`、`url`、`is_full_report` 和 `local_file`。`source_class` 只接受 `bank_ir`、`exchange`、`cninfo`、`hkex`。

脚本未返回 `status=locked` 时立即停止正式点评。不得把较早年报/中报替代目标季度后仍称“最新财报”，不得把摘要、快报、英文版片段或第三方转载标为完整报告。用户只提供摘要而完整报告暂不可得时，交付范围必须改称“摘要/快报点评”，不能套用正式财报评分。

财报预测的最小资料包改为截止日前最新完整报告、上一同类报告、上年同期和当时已发布的宏观/监管数据；目标报告属于封存资料，不在预测冻结前读取。

## 2. 来源优先级

| 优先级 | 来源 | 用途 |
|---|---|---|
| 1 | 交易所/巨潮/银行 IR 发布的正式报告与附注 | 财务数据、口径、管理层讨论 |
| 2 | 央行、金监总局、统计局、财政部等官方数据 | 宏观与监管事实 |
| 3 | 银行业绩会实录、IR 演示、管理层采访原文 | 解释驱动、指引与口径 |
| 4 | 可信数据库和券商报告 | 交叉验证、同业共识 |
| 5 | 新闻、自媒体、历史文章 | 线索与待验证假设 |

正式报告存在更正公告时，使用更正后版本并保留更正说明。

## 3. 精确配对

财报与文章/预测必须按“银行 + 报告年份 + 报告期 + A/H股口径”配对。检查：

- 银行全称和证券代码；
- 年报、半年报、一季报、三季报；
- 报告发布日期是否晚于预测文章；
- A 股中国会计准则与 H 股国际准则是否存在口径差异；
- 修订版、摘要版与完整版；
- 集团口径还是母行口径。

预测回测不得把报告发布后的信息倒灌到预测输入。历史时点任务还须使用当时可得的数据版本：记录原发布日期、修订发布日期和访问版本；若找不到 vintage 数据，明确标注“可能含事后修订”，不得声称严格先验回测。

建立输入白名单后再开始预测。白名单只包含 `publication_date <= cutoff_date` 的报告、文章和宏观发布；目标期报告、事后点评、后续管理层解释、完整回测表和包含目标实际值的方法证据全部进入封存清单。预测冻结前不要打开或全文检索封存目录。

## 4. 数据字典

每个核心数据在工作底稿保留下列字段：

| 字段 | 示例 |
|---|---|
| metric | 归母净利润 |
| value/unit | 123.45 亿元 |
| period | 20XXFY |
| scope | 集团 |
| basis | 中国会计准则 |
| cumulative_or_quarter | 累计 |
| source | 某银行20XX年年度报告 |
| page/note | 第X页 |
| status | 披露值/计算值/估计值 |
| formula | Q4=FY−9M |

输出中将标签统一为：

- `[披露]`：报告直接给出；
- `[计算]`：由披露值按明确公式计算；
- `[估计]`：依赖假设；
- `[推断]`：因果解释，存在替代解释。

正式财报点评同时使用 [earnings-review-data-dictionary.md](earnings-review-data-dictionary.md) 和内部144项指标清单完成提取与核对。默认报告不展开逐项底稿，但核心表必须直接展示关键证据、数据性质和可靠程度，C/D级估算必须展示公式、假设、区间、残差和限制。用户明确要求审计底稿、逐项追溯、完整数据附录或监管复核时，再把完整底稿内嵌到 Markdown 附录。

完成底稿后先运行 `scripts/validate_earnings_ledger.py`。正文所有数值必须来自这一份已校验底稿或由其确定公式生成，不再维护第二套手工数字。发现分项与合计、拨备覆盖率、评分范围或状态/可靠程度冲突时，先修底稿再重写正文。

## 5. 口径陷阱清单

### 净息差

- 分母是否为全部生息资产、剔除交易资产对应负债后的口径，或“还原口径”；
- 信用卡分期手续费是否重分类到利息收入；
- 报告披露的是累计年化净息差，不能直接当单季净息差；
- 日均余额与期初期末均值可能有差异。

### 非息与保险

- 投资收益、公允价值变动、汇兑损益、其他综合收益可能相互转移；
- 并表保险子公司的保费收入应与赔付、准备金和其他业务成本净看；
- 快报不披露手续费明细时，不强行把非息变化归因到某一项。

### 资产质量

- 不良率下降可能来自核销、转出、并表和分母扩张；
- 迁徙率公式、年化方式和到期率系数可能跨期变化；
- 关注率、逾期率、新生成不良与信用成本的方向应相互校验；
- 贷款减值准备与信用减值损失不是同一概念。

### 资本

- 新旧资本管理办法可能改变 RWA 和资本充足率的可比性；
- 分红在特定季度计提会影响资本充足率环比；
- 内评法与权重法银行不能只比较绝对资本充足率。

## 6. 双重验证

核心数据至少满足其一：

1. 同一正式报告的主表与附注勾稽；
2. 正式报告与交易所结构化披露一致；
3. 本期与上期累计数重构单季后，与报告分季度表一致；
4. 银行官方报告与官方业绩演示一致。

因同一报告内的两个表并非独立来源，关键结论仍应寻找第二官方披露或做跨表勾稽。差异超过 1% 时停止引用，先解释单位、范围、会计口径或修订原因。

## 7. 算术与审计

- 统一金额单位后再计算；百分点与百分比变化分开。
- 用 `scripts/bank_metrics.py` 或 Decimal/电子表格公式计算。
- 核对 `总收入 = 净利息收入 + 手续费净收入 + 其他非息收入` 的定义边界。
- 核对 `税前利润 − 所得税 ≈ 净利润`，并解释少数股东损益。
- 核对资产质量和资本的 `期初 + 变动 = 期末`。
- 对每个表执行合计、符号、单位、场景顺序和小数位检查。

## 8. 信息不足时

不要补想象。列出：已知、未知、可计算、可估计、不可估计、所需下一份披露。降低置信度并缩小结论范围。

季报未披露核销或新生成不良时，可以通过不良贷款余额变化和贷款减值准备余额变化构建情景，但必须同时满足：

1. 默认命名为“估算核销/转出”和“未披露风险净流量情景”，不冒充披露事实；
2. 展示公式、输入页码、假设、中性值、上下限和残差；
3. 只使用总信用减值、并表/汇率/资产包影响不能分离或处置流量缺失时，一律标记低（D级）；
4. 使用关注/逾期/迁徙/阶段二或细分业务中至少一类信号交叉验证；
5. 信号相反时只写“分化”，不宣布趋势拐点；
6. 后续完整披露出现时，复盘估算误差并更新方法。

只有同时满足以下三项，才可把情景区间称为“新生成不良代理下限”：

1. 转入、并表、汇率等可能增加不良余额的流量已经完整识别或给出有依据的上界；
2. 核销/转出区间由贷款减值准备变化支持，没有用全部信用减值直接冒充贷款减值；
3. 核销、批量转让、资产证券化和其他转出不存在重复计算。

反推值与核对差额由同一恒等式构造，零残差不构成独立验证。交叉验证只能来自另一张官方表、另一项风险指标或后续正式披露。

并表、汇率、重大资产包买卖或重分类无法量化时，不输出C级精确区间；降为D级宽区间或N/A。

历史文章的预测表若只以外链图片存在，先核对图片发布日期和可访问性；可访问且不越过截止日时保留原图并人工复核。不可访问时，用文章正文和白名单内官方报告重建模型，不猜原表数值，并把“无法复原原作者表格”写入限制。
